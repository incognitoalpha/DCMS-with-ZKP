# DCMS - Decentralized Cottage Management System

A blockchain dApp for booking shared cottage resources.

This version focuses on validating bookings, votes, and reputation claims with ZK-SNARKs (Groth16). Proofs are generated client-side and verified on-chain.

## What ZKP Does (And Does Not Do)

ZKP here is used as a "proof of validity" layer. For example, a booking transaction includes a Groth16 proof that is checked by the contract before it accepts the booking.

Important: ZKP does not hide the transaction sender on Ethereum. Every transaction still has a `from` address on-chain, and logs/indexers can still observe that sender. Some events in this project also include addresses. This is not full anonymity.

## Features

- Admin resource management (`addResource`, `setResourceActive`)
- ZK booking: commitment + nullifier + time range validated by a Groth16 proof, with on-chain double-booking checks
- ZK voting (demo identity set + nullifiers)
- ZK reputation claim (threshold proof)
- Activity feed built from contract events

## Quick Start (Local Hardhat Chain)

You need Node 18+ and MetaMask installed in your browser.

### 1. Install Dependencies

```bash
cd contracts && npm install
cd ../frontend && npm install
```

### 2. Start a Local Hardhat Chain

In one terminal, from `contracts/`:

```bash
npx hardhat node
```

This prints 20 funded accounts. Account #0 is the deployer/admin.

### 3. Deploy the Contract + Verifiers + Seed Resources

In a second terminal, from `contracts/`:

```bash
npx hardhat run scripts/deploy.js --network localhost
```

This deploys the verifiers and the DCMS contract, seeds sample resources, and writes:

- ABI: `frontend/src/lib/contract/DCMS.json`
- Deployment: `frontend/src/lib/contract/address.json`

The frontend imports those files directly, so no env vars are needed.

### 4. Run the Frontend

From `frontend/`:

```bash
npm run dev
```

Open `http://localhost:3000`.

### 5. Configure MetaMask

Add a custom network in MetaMask:

| Field | Value |
|---|---|
| Network name | DCMS Local |
| RPC URL | `http://127.0.0.1:8545` |
| Chain ID | `31337` |
| Currency | ETH |

Then import a Hardhat account using one of the private keys printed by `npx hardhat node`.

Note: every time you stop and restart `npx hardhat node`, the chain resets. You'll need to redeploy (step 3). If MetaMask UI looks stuck, clear activity: Settings -> Advanced -> Clear activity tab data.

## Running the Test Suite

From `contracts/`:

```bash
npx hardhat test
```

## What Is On-Chain vs Off-Chain

| Data | Where |
|---|---|
| Resources, bookings, proposals, reputation state | On-chain (contract state) |
| ZK verifiers | On-chain (verifier contracts) |
| Proofs | Generated client-side and submitted in transactions |
| Circuits and proving keys | In-repo; browser uses `frontend/public/zk/*` |

There is no backend, no database, and no indexer requirement. The Next.js app talks directly to the contract.

## Security Notes (High Level)

- Admin-only functions are gated by the `onlyAdmin` modifier.
- Double booking is prevented on-chain by checking existing bookings for a resource.
- ZK verifiers are public and deterministic (generated by snarkjs for the circuits).
